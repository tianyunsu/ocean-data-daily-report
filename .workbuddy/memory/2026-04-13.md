## 自动化任务失败原因排查（11:36）

### 问题现象
连续几天（4月10日-13日）自动化任务均失败，无日报产出。

### 根本原因：登录会话在任务执行时段失效

**完整失败链条（日志精确还原）：**

1. **10:29:00** WorkBuddy启动，调度引擎检测到任务超时，立即触发"海洋AI研究日报"执行
2. **10:29:00.571** `[SchedulerEngine] Automation "海洋AI研究日报" (ai) is due`
3. **10:29:00.573** `[warning] Queueing automation execution until Agent Manager bridge is ready`（等待渲染层就绪）
4. **10:29:28** Agent Manager 就绪，开始执行任务，路由到 Claw window 3，创建新会话
5. **10:29:34** Session刷新成功（正常）
6. **10:29:39** ⚠️ **`[IDE-NATIVE-AUTH] Logout started`** → **用户会话被强制登出！**
7. **10:29:39.690** `Auth userId changed: 99319ad6... -> none` → 会话变为未登录状态
8. **10:29:42** WorkBuddy 弹出浏览器跳转登录页（需人工干预）
9. **10:30:28** `[error] Sub-run run-1776047340571-1:1 failed: **Timeout waiting for response from __new__**`（任务在未登录状态超时）
10. **10:30:28** `success=false, forcedError=none` → 任务失败
11. **10:34:42** `[error] Login failed: Authentication timeout after 5 minutes`（自动登录5分钟超时）

### 根本原因结论
**登录会话（Token）在自动化任务执行期间过期/失效，触发强制登出流程，导致任务在无认证状态下运行超时失败。**

具体表现：
- WorkBuddy 在启动后约39秒就发生了 Logout（session被踢出）
- 任务已经开始创建新会话（`__new__`），但在等待响应期间认证失效
- 系统随即弹出需要人工操作的浏览器登录页
- 无人值守状态下5分钟登录超时，任务失败
- 下次运行前若不手动重新登录，下一次调度仍会失败

### 为何连续几天失败
每天9:00触发任务时，如果 WorkBuddy 当时正好处于未登录或 token 刚过期状态，就会进入同样的失败循环。连续几天的失败说明 token 的有效期约为1天，每天触发时都恰好处于未登录状态。

### 解决方案

**短期措施（今日）：**
1. 手动重新登录 WorkBuddy，确保 session 有效
2. 手动触发今天的日报

**长期根治方案（需用户操作）：**
1. WorkBuddy 本身需要保持长期登录状态：每次使用后不要手动登出
2. 确认是否有"保持登录"/"记住我"选项 → 启用
3. 如有 Token 自动刷新机制，确认其是否正常工作
4. 考虑设置开机自启，让 WorkBuddy 在9:00前已经处于登录状态
